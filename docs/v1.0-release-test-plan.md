## Track 1: Upgrade on AWS (existing instance)

Current state: v0.4 with per-db backups already running.

### 1. Install the release binary

```bash
curl -fsSL https://github.com/danbiagini/FreeDB/releases/download/v1.0-rc1/freedb-linux-amd64 \
  -o /tmp/freedb && chmod +x /tmp/freedb
/tmp/freedb --version   # should show v1.0-rc1
```

### 2. Upgrade

```bash
sudo cp /tmp/freedb /usr/local/bin/freedb
sudo freedb upgrade --dry-run    # preview pending migrations
sudo freedb upgrade              # run it
cat /etc/freedb/version          # should show v0.5
```

### 3. Verify existing apps still work

- [ ] `sudo freedb list` — all apps present
- [ ] `sudo freedb status aifa-api-stage` — shows detail
- [ ] Hit the app's domain in a browser — responds correctly
- [ ] TUI dashboard shows resource summary (mem, CPU, disk)

### 4. Database management

- [ ] TUI: `[D]` — databases list shows all DBs with backup status
- [ ] Select a DB — shows local path and cloud URL
- [ ] `[a]` — create a test database, verify name input works
- [ ] `[d]` — drop the test database

### 5. Backup and restore

```bash
sudo bash -c '. /opt/freedb/backup.env && /opt/freedb/backup-db.sh'
```

- [ ] All databases backed up individually (not pg_dumpall)
- [ ] Files have UTC timestamp: `dbname_YYYYMMDD_HHMMSSZ.sql.gz`
- [ ] Cloud upload succeeds to correct S3 bucket
- [ ] `cat /var/lib/freedb/backup-status.json` — has `databases` array with `cloud_url`

### 6. Restore

```bash
sudo freedb restore aifa-api-stage              # list backups
sudo freedb restore aifa-api-stage <timestamp>  # restore + restart app
```

- [ ] Lists available backups
- [ ] Restore succeeds, app container restarts
- [ ] App still works after restore

### 7. ACME email

```bash
sudo freedb acme-email                        # shows current
sudo freedb acme-email test@example.com       # updates
sudo freedb acme-email                        # confirms change
```

### 8. Add a system container (no Traefik)

- [ ] TUI: `[a]` -> name: `test-sys` -> image: `ubuntu/24.04/cloud` -> Expose via Traefik: `n` -> DB: `n`
- [ ] Container created, no Traefik route
- [ ] `sudo freedb destroy test-sys --yes` — clean removal

## Track 2: Fresh install on GCP

Start with a clean Debian 12 VM + attached persistent disk.

### 1. Provision infrastructure

```bash
cd infra
tofu workspace new v1-test
tofu apply -var-file=test.tfvars
```

### 2. Install the platform

```bash
ssh <gcp-instance>
curl -fsSL https://raw.githubusercontent.com/danbiagini/FreeDB/main/install.sh | bash
# Reboot if prompted for ZFS, then re-run
```

- [ ] Install completes without errors
- [ ] Prompted for Let's Encrypt email
- [ ] Prompted for backup bucket name

### 3. Install the TUI

```bash
curl -fsSL https://github.com/danbiagini/FreeDB/releases/download/v1.0-rc1/freedb-linux-amd64 \
  -o /usr/local/bin/freedb && chmod +x /usr/local/bin/freedb
```

### 4. Health check

```bash
sudo freedb check
```

- [ ] All checks pass

### 5. Deploy an OCI app with Traefik + DB

- [ ] TUI: `[a]` -> name, docker image, expose: `y`, domain, port, TLS: `y`, DB: `y`
- [ ] App accessible at `https://domain`
- [ ] Certificate issued by Let's Encrypt
- [ ] Database created and `DATABASE_URL` injected

### 6. Deploy a system container without Traefik

- [ ] TUI: `[a]` -> name, `debian/12/cloud`, expose: `n`, DB: `n`

### 7. Backup cycle

```bash

```

- [ ] Per-database backup files created
- [ ] Cloud upload to GCS bucket

### 8. Restore cycle

- [ ] TUI: `[D]` -> select DB -> `[r]` -> select backup -> confirm

### 9. Upgrade path (simulate)


- [ ] v0.5 migration runs (installs backup script)
- [ ] Version file updated

- [ ] `tofu destroy` — infrastructure torn down
